Favor object composition over class inheritance.

As a researcher with a background in machine learning and robotics. Most of my time and effort have been focused on optimizing the benchmark performance of an algorithm for a specific problem for the sake of publishing a paper. It is a shame that despite that I have been using Python for my research projects for a long time, I never really learn the design pattern for Python. This blog is a record of my learning journey.

According to the talk by Brandon Rhodes, there are three stages of everyone's programming journey.

  1. Procedural -- All code in main()
  2. Abstraction -- Functions
  3. Encapsulation -- Classes
For machine learning researchers, it is totally fine to just stay in the first stage, where you could implement all you code in main() with some level of abstraction using functions and classes. You can still achieve great results. However, the most interesting part of programming is the third stage -- Encapsulation. It is the stage where you start to think about the Design.

According to the book Design Patterns: Elements of Reusable Object-Oriented Software, there are 23 design patterns in total. However, the book was published in 1994, and some of the patterns are not very useful in modern software development, especially for python language. According to Brandon Rhodes and his talk on The Classical Design Patterns: Where Are They Now?, most of us have been fooled. In his talk, he made the following conclusion:
  • 6 patterns: useless if you have 1st class functions
  • 3 patterns: are anti-patterns that obscure our data
  • 4 patterns: decompose big classes, but are obscure
  • 1 pattern: ambitious but fatally limited
  • 2 patterns: language built-ins
  • 3 patterns: small but useful
  • 4 patterns: huge and dominate our frameworks

Composite: Compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.
Structure

    • Python example: a simple filesystem tree where folders contain files or other folders.

    class Component:
    def render(self, indent=0):
        raise NotImplementedError

class File(Component):
    def __init__(self, name):
        self.name = name

    def render(self, indent=0):
        print(" " * indent + f"- {self.name}")

class Folder(Component):
    def __init__(self, name):
        self.name = name
        self.children = []

    def add(self, *components):
        self.children.extend(components)
        return self

    def render(self, indent=0):
        print(" " * indent + f"+ {self.name}/")
        for c in self.children:
            c.render(indent + 2)

# Build a small tree
root = Folder("project")
src = Folder("src").add(File("main.py"), File("utils.py"))
assets = Folder("assets").add(File("logo.png"))
root.add(src, assets, File("README.md"))
root.render()

    Clients call render() without caring whether an object is a File (leaf) or Folder (composite). The composite delegates the work to its children.

    ðŸ“Œ The original 23 Design Patterns by Gang of Four

    Category Design Patterns
    Creational Singleton: Ensure a class has only one instance and provide a global point of access to it.
    Factory Method: Define an interface for creating an object, but let subclasses decide which class to instantiate.
    Abstract Factory: Provide an interface for creating families of related or dependent objects without specifying their concrete classes.
    Builder: Separate the construction of a complex object from its representation so that the same construction process can create different representations.
    Prototype: Create new objects by cloning existing objects.
    Structural Adapter: Convert the interface of a class into another interface clients expect. Adapter lets classes work together that couldn't otherwise because of incompatible interfaces.
    Bridge: Decouple an abstraction from its implementation so that the two can vary independently.
    Composite: Compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.
    Decorator: Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.
    Facade: Provide a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use.
    Flyweight: Flyweight is a structural design pattern that provides ways to decrease object count thus reducing memory usage by sharing object instances.
    Proxy: Provide a surrogate or placeholder for another object to control access to it.
    Behavioral Chain of Responsibility: Chain the receiving objects and pass the request along the chain until an object handles it.
    Command: Encapsulate a request as an object, thereby allowing for parameterization of clients with different requests, and support undoable operations.
    Interpreter: Given a language, define a representation for its grammar along with an interpreter that uses the representation to interpret sentences in the language.
    Iterator: Provide a way to access the elements of an object sequentially without exposing its underlying representation.
    Mediator: Define an object that encapsulates how a set of objects interact. Mediator promotes loose coupling by keeping objects from referring to each other explicitly, and it lets you vary their interaction independently.
    Memento: Capture and restore an object's internal state without violating encapsulation.
    Observer: Define a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.
    State: Allow an object to alter its behavior when its internal state changes. The object will appear to change its class.
    Strategy: Define a family of algorithms, encapsulate each one, and make them interchangeable. Strategy lets the algorithm vary independently from clients that use it.
    Template Method: Define the skeleton of an algorithm in an operation, deferring some steps to subclasses. Template Method lets subclasses redefine certain steps of an algorithm without changing the algorithm's structure.
    Visitor: Represent an operation to be performed on the elements of an object structure. Visitor lets you define a new operation without changing the classes of the elements on which it operates.
