Favor object composition over class inheritance.

As a researcher with a background in machine learning and robotics. Most of my time and effort have been focused on optimizing the benchmark performance of an algorithm for a specific problem for the sake of publishing a paper. It is a shame that despite that I have been using Python for my research projects for a long time, I never really learn the design pattern for Python. This blog is a record of my learning journey.

According to the talk by Brandon Rhodes, there are three stages of everyone's programming journey.

  1. Procedural -- All code in main()
  2. Abstraction -- Functions
  3. Encapsulation -- Classes
For machine learning researchers, it is totally fine to just stay in the first stage, where you could implement all you code in main() with some level of abstraction using functions and classes. You can still achieve great results. However, the most interesting part of programming is the third stage -- Encapsulation. It is the stage where you start to think about the Design.

üìå Definition

Let's begin with the definition of design pattern.
Design Pattern: A design pattern is a general, reusable solution to a commonly occurring problem within a given context in software design. It is a description of a problem that occurs frequently in software design, and a solution to that problem.

üìå The Catalog of Design Patterns

According to the book Design Patterns: Elements of Reusable Object-Oriented Software, there are 23 design patterns in total. However, the book was published in 1994, and some of the patterns are not very useful in modern software development, especially for python language. According to Brandon Rhodes and his talk on The Classical Design Patterns: Where Are They Now?, most of us have been fooled. In his talk, he made the following conclusion:
  • 6 patterns: useless if you have 1st class functions
  • 3 patterns: are anti-patterns that obscure our data
  • 4 patterns: decompose big classes, but are obscure
  • 1 pattern: ambitious but fatally limited
  • 2 patterns: language built-ins
  • 3 patterns: small but useful
  • 4 patterns: huge and dominate our frameworks

üìè Composite

Compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.

In short: separate the construction steps from the final object so the same sequence can build different variants.

Structure
  • Component ‚Äì defines common operations.
  • Leaf ‚Äì basic object with real behavior.
  • Composite ‚Äì container that holds children and delegates operations.

    • Python example: a simple filesystem tree where folders contain files or other folders.

    class Component:
    def render(self, indent=0):
        raise NotImplementedError

class File(Component):
    def __init__(self, name):
        self.name = name

    def render(self, indent=0):
        print(" " * indent + f"- {self.name}")

class Folder(Component):
    def __init__(self, name):
        self.name = name
        self.children = []

    def add(self, *components):
        self.children.extend(components)
        return self

    def render(self, indent=0):
        print(" " * indent + f"+ {self.name}/")
        for c in self.children:
            c.render(indent + 2)

# Build a small tree
root = Folder("project")
src = Folder("src").add(File("main.py"), File("utils.py"))
assets = Folder("assets").add(File("logo.png"))
root.add(src, assets, File("README.md"))
root.render()

    Clients call render() without caring whether an object is a File (leaf) or Folder (composite). The composite delegates the work to its children.

    üìè Chain of Responsibility

    Avoid coupling the sender of a request to its receiver by giving more than one object a chance to handle the request. Chain the receiving objects and pass the request along the chain until an object handles it.

    In short: wrap an incompatible object so it looks like the interface your code expects.

    Structure
  • Handler ‚Äì defines an interface for handling requests.
  • Concrete Handler ‚Äì implements the successor link.
  • Client ‚Äì initiates requests.
  • Request ‚Äì the object being handled.

    • Python example: a simple logger with multiple handlers.

    class Handler:
    def __init__(self, successor=None):
        self.successor = successor

    def handle(self, request):
        if self.successor:
            return self.successor.handle(request)
        return f"No handler could process: {request}"

class ChatbotHandler(Handler):
    def handle(self, request):
        if request == "reset password":
            return "Chatbot: Reset link sent to email."
        return super().handle(request)

class JuniorAgentHandler(Handler):
    def handle(self, request):
        if request == "refund":
            return "Junior Agent: Refund request approved."
        return super().handle(request)

class SeniorAgentHandler(Handler):
    def handle(self, request):
        if request == "technical issue":
            return "Senior Agent: Technical issue resolved."
        return super().handle(request)

# Build chain: Chatbot ‚Üí Junior Agent ‚Üí Senior Agent
handler_chain = ChatbotHandler(JuniorAgentHandler(SeniorAgentHandler()))

# Test
print(handler_chain.handle("reset password"))   # Chatbot handles
print(handler_chain.handle("refund"))           # Junior Agent handles
print(handler_chain.handle("technical issue"))  # Senior Agent handles
print(handler_chain.handle("complaint"))        # No handler could process

    üìè Command

    Encapsulate a request as an object, thereby letting you parameterize clients with different requests, queue or log requests, and support undoable operations.

    In short: share heavy, immutable data (intrinsic state) across many lightweight objects; pass varying details (extrinsic state) at use time.

    Structure
  • Command ‚Äì defines an interface for executing an operation.
  • Concrete Command ‚Äì implements the command interface.
  • Invoker ‚Äì initiates requests.
  • Receiver ‚Äì the object that performs the action.

    • Python example: a simple command pattern.

    ## -- Command Interface -- ##
class Command:
    def execute(self):
        raise NotImplementedError

## -- Receiver (does the real work) -- ##
class Light:
    def turn_on(self):
        return "Light is ON"

    def turn_off(self):
        return "Light is OFF"

## -- Concrete Command -- ##
class TurnOnLightCommand(Command):
    def __init__(self, light):
        self.light = light

    def execute(self):
        return self.light.turn_on()

class TurnOffLightCommand(Command):
    def __init__(self, light):
        self.light = light

    def execute(self):
        return self.light.turn_off()

## -- Invoker (calls the command) -- ##
class RemoteControl:
    def __init__(self):
        self.commands = []

    def submit(self, command):
        self.commands.append(command)
        return command.execute()
        
## -- Usage -- ##
# Receiver
light = Light()
# Commands
turn_on = TurnOnLightCommand(light)
turn_off = TurnOffLightCommand(light)
# Invoker
remote = RemoteControl()
print(remote.submit(turn_on))   # Light is ON
print(remote.submit(turn_off))  # Light is OFF

    üìè Interpreter

    Given a language, define a represention for its grammar along with an interpreter that uses the representation to interpret sentences in the language.
    Python example: 
    ## -- Abstract Expression -- ##
class Expression:
    def interpret(self):
        raise NotImplementedError

## -- Terminal Expression (basic unit) -- ##
class Number(Expression):
    def __init__(self, value):
        self.value = value

    def interpret(self):
        return self.value

## -- Non-Terminal Expressions (combine rules) -- ##
class Add(Expression):
    def __init__(self, left, right):
        self.left = left
        self.right = right

    def interpret(self):
        return self.left.interpret() + self.right.interpret()


class Multiply(Expression):
    def __init__(self, left, right):
        self.left = left
        self.right = right

    def interpret(self):
        return self.left.interpret() * self.right.interpret()


## -- Example usage -- ##
if __name__ == "__main__":
    # Represents: (5 + 3) * 2
    expression = Multiply(
        Add(Number(5), Number(3)),
        Number(2)
    )
    print(expression.interpret())  # Output: 16

    üìè Iterator

    Provide a way to access the elements of an aggregate object sequentially without exposing its underlying representation.

    In short: hide traversal logic inside an iterator so clients use a uniform loop.

    Python example: a custom iterable with its own iterator.

    class MyIterator:
    def __init__(self, items):
        self._items = list(items)
        self._i = 0
    def __iter__(self):
        return self
    def __next__(self):
        if self._i >= len(self._items):
            raise StopIteration
        val = self._items[self._i]
        self._i += 1
        return val

class MyCollection:
    def __init__(self, *items):
        self._items = items
    def __iter__(self):
        return MyIterator(self._items)

nums = MyCollection(1, 2, 3)
print([x for x in nums])  # [1, 2, 3]

    üìè Visitor

    Represent an operation to be performed on the elements of an object structure. Visitor lets you define a new operation without changing the classes of the elements.

    In short: add new operations over a stable element hierarchy without modifying element classes.

    Python example: compute sizes over a small filesystem-like tree using a visitor.

    class Visitor:
    def visit_file(self, file):
        raise NotImplementedError
    def visit_folder(self, folder):
        raise NotImplementedError

class SizeVisitor(Visitor):
    def __init__(self):
        self.total = 0
    def visit_file(self, file):
        self.total += file.size
    def visit_folder(self, folder):
        for child in folder.children:
            child.accept(self)

class Element:
    def accept(self, visitor: Visitor):
        raise NotImplementedError

class File(Element):
    def __init__(self, name, size):
        self.name, self.size = name, size
    def accept(self, visitor):
        return visitor.visit_file(self)

class Folder(Element):
    def __init__(self, name, *children):
        self.name = name
        self.children = list(children)
    def accept(self, visitor):
        return visitor.visit_folder(self)

root = Folder("root",
    File("a.txt", 10),
    Folder("docs", File("b.pdf", 50), File("c.md", 5))
)

v = SizeVisitor()
root.accept(v)
print(v.total)  # 65

    üìå Three Small but Useful Patterns

    üìè Builder

    Separate the construction steps from the final object so the same sequence can build different variants.
    Structure
  • Director ‚Äì constructs an object using the Builder interface.
  • Builder ‚Äì provides an interface for creating parts of a Product object.
  • Concrete Builder ‚Äì implements the Builder interface and builds a Product.
  • Product ‚Äì the object being built.
    • Python example: 
    ## -- Product -- ##
class Car:
    def __init__(self):
        self.parts = []

    def add_part(self, part):
        self.parts.append(part)

    def __str__(self):
        return f"Car with parts: {', '.join(self.parts)}"

# Builder
class CarBuilder:
    def __init__(self):
        self.reset()

    def reset(self):
        self._car = Car()

    def add_wheels(self):
        self._car.add_part("4 wheels")
        return self

    def add_engine(self):
        self._car.add_part("engine")
        return self

    def add_doors(self):
        self._car.add_part("4 doors")
        return self

    def build(self):
        car = self._car
        self.reset()
        return car

# Director (defines *what kind* of car to build)
class Director:
    def __init__(self, builder: CarBuilder):
        self._builder = builder

    def build_sports_car(self):
        return (
            self._builder.reset()
            .add_wheels()
            .add_engine()
            .add_doors()
            .build()
        )

    def build_simple_car(self):
        return (
            self._builder.reset()
            .add_wheels()
            .add_engine()
            .build()
        )

# Usage
if __name__ == "__main__":
    builder = CarBuilder()
    director = Director(builder)
    sports_car = director.build_sports_car()
    simple_car = director.build_simple_car()
    print(sports_car)   # Car with parts: 4 wheels, engine, 4 doors
    print(simple_car)   # Car with parts: 4 wheels, engine

    üìè Adapter

    Convert the interface of a class into another interface clients expect. Adapter lets classes work together that couldn't otherwise because of incompatible interfaces.
    Structure
  • Target ‚Äì the interface clients expect.
  • Adapter ‚Äì adapts the Adaptee to the Target interface.
  • Adaptee ‚Äì the object with the incompatible interface.
    • Python example: 
    ## -- Target (what the client expects) -- ##
class Target:
    def request(self):
        return "Target: The default request"

## -- Adaptee (has a different interface) -- ##
class Adaptee:
    def specific_request(self):
        return "Adaptee:  û…πo ç sdrawkcab"  # returns reversed text for demo

## -- Adapter (makes Adaptee compatible with Target) -- ##
class Adapter(Target):
    def __init__(self, adaptee: Adaptee):
        self.adaptee = adaptee

    def request(self):
        # Convert Adaptee's output into what Target expects
        return f"Adapter: (TRANSLATED) {self.adaptee.specific_request()[::-1]}"


# Client code
if __name__ == "__main__":
    target = Target()
    print(target.request())  # Target: The default request

    adaptee = Adaptee()
    print(adaptee.specific_request())  # Adaptee:  û…πo ç sdrawkcab

    adapter = Adapter(adaptee)
    print(adapter.request())  # Adapter: (TRANSLATED) backwards work

    üìè Flyweight

    Flyweight is a structural design pattern that provides ways to decrease object count thus reducing memory usage by sharing object instances.
    Structure
  • FlyweightFactory ‚Äì creates and manages flyweight objects.
  • Flyweight ‚Äì the shared intrinsic state.
  • Client ‚Äì the objects with extrinsic state.
    • Python example: 
    ## -- Flyweight -- ##
class Character:
    def __init__(self, symbol, font):
        self.symbol = symbol    # intrinsic (shared)
        self.font = font        # intrinsic (shared)

    def display(self, position):
        # position is extrinsic (unique per use)
        print(f"Character: {self.symbol}, Font: {self.font}, Position: {position}")

## -- Flyweight Factory -- ##
class CharacterFactory:
    def __init__(self):
        self._characters = {}

    def get_character(self, symbol, font):
        key = (symbol, font)
        if key not in self._characters:
            self._characters[key] = Character(symbol, font)
        return self._characters[key]

# Client code
if __name__ == "__main__":
    factory = CharacterFactory()

    # Reuses flyweight objects for repeated chars
    c1 = factory.get_character("A", "Arial")
    c2 = factory.get_character("A", "Arial")
    c3 = factory.get_character("B", "Arial")

    c1.display(1)  # Character: A, Font: Arial, Position: 1
    c2.display(2)  # Character: A, Font: Arial, Position: 2
    c3.display(3)  # Character: B, Font: Arial, Position: 3

    # c1 and c2 are the same object (shared flyweight)
    print(c1 is c2)  # True

    üìå The original 23 Design Patterns by Gang of Four

    Category Design Patterns
    Creational Singleton: Ensure a class has only one instance and provide a global point of access to it.
    Factory Method: Define an interface for creating an object, but let subclasses decide which class to instantiate.
    Abstract Factory: Provide an interface for creating families of related or dependent objects without specifying their concrete classes.
    Builder: Separate the construction of a complex object from its representation so that the same construction process can create different representations.
    Prototype: Create new objects by cloning existing objects.
    Structural Adapter: Convert the interface of a class into another interface clients expect. Adapter lets classes work together that couldn't otherwise because of incompatible interfaces.
    Bridge: Decouple an abstraction from its implementation so that the two can vary independently.
    Composite: Compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.
    Decorator: Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.
    Facade: Provide a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use.
    Flyweight: Flyweight is a structural design pattern that provides ways to decrease object count thus reducing memory usage by sharing object instances.
    Proxy: Provide a surrogate or placeholder for another object to control access to it.
    Behavioral Chain of Responsibility: Chain the receiving objects and pass the request along the chain until an object handles it.
    Command: Encapsulate a request as an object, thereby allowing for parameterization of clients with different requests, and support undoable operations.
    Interpreter: Given a language, define a representation for its grammar along with an interpreter that uses the representation to interpret sentences in the language.
    Iterator: Provide a way to access the elements of an object sequentially without exposing its underlying representation.
    Mediator: Define an object that encapsulates how a set of objects interact. Mediator promotes loose coupling by keeping objects from referring to each other explicitly, and it lets you vary their interaction independently.
    Memento: Capture and restore an object's internal state without violating encapsulation.
    Observer: Define a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.
    State: Allow an object to alter its behavior when its internal state changes. The object will appear to change its class.
    Strategy: Define a family of algorithms, encapsulate each one, and make them interchangeable. Strategy lets the algorithm vary independently from clients that use it.
    Template Method: Define the skeleton of an algorithm in an operation, deferring some steps to subclasses. Template Method lets subclasses redefine certain steps of an algorithm without changing the algorithm's structure.
    Visitor: Represent an operation to be performed on the elements of an object structure. Visitor lets you define a new operation without changing the classes of the elements on which it operates.
